# SentinelFit Operations: Architecture & Logic Clarifications

As a senior developer, I have reviewed the SentinelFit prompt to identify potential implementation gaps. The following 15 questions address ambiguities that would significantly affect the database schema, security posture, and system performance.

### 1. The HMAC Secret Storage Dilemma
* **What sounded ambiguous:** The requirement specifies HMAC-SHA256 request signing via the Vue.js frontend to prevent replay attacks.
* **How it was understood:** Each client (browser) needs a secret key to sign requests.
* **How it was solved:** Since the frontend is client-side code, storing a "static" secret is insecure (any user can inspect the source). The system will implement a **Session-Specific Secret** generated by the server upon successful login (handshake), stored only in memory (Vuex/Pinia), and rotated per session.

### 2. Encryption Key Rotation Strategy
* **What sounded ambiguous:** "Keys rotated every 180 days."
* **How it was understood:** Existing encrypted data needs to remain readable.
* **How it was solved:** We will implement **Key Versioning**. The database will store a `key_id` alongside the encrypted blob. The system will maintain a "Key Vault" (local file-based for offline) containing the current and historical keys to ensure data isn't lost during rotation, rather than re-encrypting the entire multi-terabyte database every 180 days.

### 3. Face Deduplication Scope
* **What sounded ambiguous:** "Deduplication using a local similarity threshold."
* **How it was understood:** When a new face is enrolled, we check if it already exists.
* **How it was solved:** The check must include **Deactivated** accounts. If a member is deactivated but their face data is retained for 7 years (per audit rules), the system must prevent them from re-enrolling as a "new" member to bypass bans or gym fees.

### 4. PIN Re-entry vs. Session Expiry
* **What sounded ambiguous:** "Optional 4–6 digit PIN after a successful password login... sessions expire after 30 minutes."
* **How it was understood:** The PIN is a convenience feature for "warm" workstations.
* **How it was solved:** The 30-minute inactivity timer applies to the *Full Session*. However, a separate **PIN Lock Timer** (e.g., 5 minutes) will trigger a UI overlay requiring only the PIN. If the full 30 minutes pass, the PIN is invalidated, and a full password login is required.

### 5. Liveness Challenge Validation
* **What sounded ambiguous:** "Simple two-step liveness challenge requiring two timed captures with a prompted head turn."
* **How it was understood:** The system captures two frames.
* **How it was solved:** To prevent a user from just holding up a photo, the backend must validate that the **biometric landmarks** (nose tip position, eye alignment) have shifted significantly between the "Center" and "Turn" frames, ensuring the movement matches the prompt's direction (Left/Right).

### 6. The "Offline" Shared Folder Path
* **What sounded ambiguous:** "Downloadable file stored on a designated shared folder path."
* **How it was understood:** The server saves reports to a network drive.
* **How it was solved:** Since the backend is Express, we need to define the **File Protocol**. We will implement a configurable SMB/UNC path mapping in the `.env` file, ensuring the service account running the Express process has "Write" permissions to that specific network volume.

### 7. Dashboard Builder Permissions
* **What sounded ambiguous:** "Administrators use a configurable dashboard builder with drag-and-drop charts."
* **How it was understood:** Admins can move widgets.
* **How it was solved:** We will implement a **Layout Metadata Schema** in MySQL. Instead of hardcoding dashboards, the `users` or `roles` table will link to a JSON blob defining the layout (X, Y coordinates and ChartID) so each admin can have a personalized view without affecting others.

### 8. Face Capture "Quality Gates" Location
* **What sounded ambiguous:** "Receiving immediate on-screen feedback for quality gates."
* **How it was understood:** The user sees if the photo is good or bad.
* **How it was solved:** To minimize server load, **Initial Gates** (blur, face-in-frame) will be processed in the browser using `TensorFlow.js`. The **Final Gate** (similarity/deduplication) will be processed on the Express backend for security and accuracy.

### 9. Content View Attribution
* **What sounded ambiguous:** "Content views by source channel (e.g., lobby kiosk, front-desk terminal)."
* **How it was understood:** Tracking who looks at what.
* **How it was solved:** Since users may not be "logged in" when viewing a kiosk, the frontend must pass a **DeviceID** or **StationToken** in the header of the analytics API call to differentiate the "Lobby" from the "Front Desk."

### 10. Audit Trail Immutability
* **What sounded ambiguous:** "Immutable audit trail retention for 7 years."
* **How it was understood:** Records can't be deleted.
* **How it was solved:** We will implement a **Database Trigger** that prevents `UPDATE` or `DELETE` commands on the `audit_logs` table. Any attempt to modify these logs will result in a high-priority "System Integrity Anomaly" alert on the Admin Console.

### 11. Role Hierarchy and Navigation
* **What sounded ambiguous:** "Clear hierarchy and role-based navigation."
* **How it was understood:** Coaches see more than Members.
* **How it was solved:** We will implement **Inclusive Permissions**. The "Admin" role inherits all "Coach" permissions, and "Coach" inherits all "Member" permissions. Navigation links will be filtered via a `v-if="user.role >= REQUIRED_LEVEL"` directive in the Vue sidebar.

### 12. "Deactivation" vs. Search Trends
* **What sounded ambiguous:** "Deactivation instead of hard delete."
* **How it was understood:** Members are flagged as `inactive`.
* **How it was solved:** Analytics queries (search trends, views) must explicitly filter for `user.status = 'active'` unless the Coach specifically toggles "Include Historical Data," to prevent skewed performance metrics from former members.

### 13. API Rate Limiting vs. Kiosks
* **What sounded ambiguous:** "60 requests per minute per IP."
* **How it was understood:** Standard security to prevent DDOS.
* **How it was solved:** Since multiple kiosks might share a single Local Area Network (LAN) IP, we will implement **Identity-Based Rate Limiting**. Authenticated users/stations get a higher threshold, while unauthenticated "Sign-in" page requests remain strictly capped at 60.

### 14. Subscription Delivery Failure
* **What sounded ambiguous:** "Subscriptions delivered as an in-app report inbox."
* **How it was understood:** Reports show up in the app.
* **How it was solved:** We need a **Background Worker (Cron)**. If the "Shared Folder Path" is unreachable at 6:00 AM on Monday, the system will still deliver the report to the "In-App Inbox" but flag a "Storage Sync Error" to the admin to prevent silent data loss.

### 15. The "Restore Workflow" Endpoint
* **What sounded ambiguous:** "...support tested restore workflows to a [cut off]."
* **How it was understood:** The system needs a way to recover from failure.
* **How it was solved:** We will assume the target is a **Standby Recovery Instance**. The workflow will include a "Dry Run" mode that validates the backup's checksum and AES-256 decryption key before attempting to overwrite the current MySQL production schema.
